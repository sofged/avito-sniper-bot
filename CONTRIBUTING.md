# Development Guide & Quality Standards

## Local Setup & Testing
Для начала работы с проектом выполните следующие шаги:

1. Клонируйте репозиторий:
   ```bash
   git clone <repository-url>
   ```
2. Установите необходимые зависимости Python:
   ```bash
   pip install playwright openpyxl
   ```
3. Установите браузеры, необходимые для работы [[avito_parser.py]]:
   ```bash
   playwright install
   ```
4. Запуск парсера:
   ```bash
   python avito_parser.py
   ```
5. Проверка результатов:
   ```bash
   python check_results.py
   ```

## Hotspots & Fragile Zones
Следующие файлы являются критическими участками системы, требующими повышенного внимания при внесении изменений:

- [[avito_parser.py]]: Высокая сложность и центральная роль в логике сбора данных делают этот файл основным местом возникновения ошибок при изменении структуры парсинга.
- [[check_results.py]]: Прямая зависимость от формата выходного файла, создаваемого [[avito_parser.py]], делает его хрупким при изменении схемы данных Excel.
- [[.gitignore]]: Некорректная настройка может привести к коммиту чувствительных данных или временных файлов в репозиторий.

## Security Policies & Requirements
- **Запрещено**: Хардкодить персональные данные, API-ключи или прокси-серверы в коде [[avito_parser.py]]. Используйте переменные окружения.
- **Обход блокировок**: Строго соблюдайте осторожность с частотой запросов. Любые изменения в логике `asyncio` или `playwright` в [[avito_parser.py]], увеличивающие нагрузку на целевой ресурс, должны проходить обязательное ревью.
- **Инъекции**: При работе с внешними данными (описания объявлений, заголовки) всегда проводите санитарную обработку перед записью в [[avito_parser.py]] или чтением в [[check_results.py]].

## PR Quality Standards & Review Gates
Все Pull Request должны соответствовать следующим критериям:

1. **Документация**: Если изменяется логика парсинга в [[avito_parser.py]], необходимо обновить раздел "Фишки" или "Использование" в [[README.md]].
2. **Код**:
   - Отсутствие "магических чисел" — выносите параметры (как `PRICE_MIN`, `PRICE_MAX` в [[avito_parser.py]]) в конфигурационные константы.
   - Минимизация ответственности: если функция в [[avito_parser.py]] превышает 50 строк, она должна быть разбита.
3. **Тестирование**:
   - Любое изменение в [[avito_parser.py]] должно сопровождаться ручной проверкой через запуск `python check_results.py` для подтверждения корректности генерации и чтения файла.
4. **Именование веток**: Используйте префиксы `feature/`, `fix/` или `refactor/` (например, `fix/avito-parser-timeout`).

## Ownership Notes & Reviewers
Ввиду отсутствия формальных команд, текущая структура ответственности распределена следующим образом:

- **Ядро парсинга ([[avito_parser.py]])**: Требует обязательного ревью на предмет эффективности запросов и корректности обработки исключений.
- **Аналитика и вывод ([[check_results.py]])**: Требует проверки на совместимость с текущей версией структуры данных Excel.
- **Инфраструктура и документация ([[README.md]], [[.gitignore]])**: Общие зоны ответственности, любые правки должны быть прозрачны для всех контри для всех участников.
